# Nexus Operations вҖ” Architecture

## System overview

Nexus Operations is a Next.js 15 App Router application deployed on Vercel. Homeowners submit documented service requests; verified contractors claim or are auto-matched to those requests; invoices are generated by contractors and paid by homeowners through Stripe. The platform also handles contractor subscriptions and Stripe Connect onboarding for direct payouts.

Four actor roles exist: **homeowner**, **contractor**, **property-manager**, and **admin**.

**Middleware** (`middleware.ts` + `lib/supabase/proxy.ts`) runs on every non-static request:
1. Refreshes the Supabase session cookie (best-effort; read-only in server components is fine because the middleware handles refresh).
2. Redirects unauthenticated users away from `/dashboard/*` to `/auth/login`.
3. Redirects authenticated users away from auth pages (`/auth/login`, `/auth/sign-up`, `/auth/forgot-password`) to their dashboard.
4. Applies HTTP security headers on every response.

Role-scoped path enforcement (e.g., preventing a contractor from reaching `/dashboard/homeowner/*`) is **not** done in middleware вҖ” it is enforced at the route and component level.

## Trust boundaries

### Browser вҶ’ API

Every request that touches user data must arrive with a valid Supabase session cookie. API routes call `createClient()` (anon key + cookie) and call `supabase.auth.getUser()` before reading or writing. Requests without a valid session receive `401 Unauthorized`.

User-controlled input (request body, query parameters) is the primary trust boundary. Each route is responsible for validating its own inputs. Zod schemas are used in the form layer; API routes perform their own checks.

### API вҶ’ Supabase (two client types)

| Client | Key used | RLS | When to use |
|---|---|---|---|
| `createClient()` from `lib/supabase/server.ts` | Anon key + cookie | **Enforced** | All user-facing reads/writes |
| `getAdminClient()` from `lib/supabase/admin.ts` | `SUPABASE_SERVICE_ROLE_KEY` | **Bypassed** | See В§RLS bypass inventory |

The admin client is a module-level singleton initialized lazily. It is never imported from client-side code.

### Stripe вҶ’ API (inbound webhooks)

`/api/stripe/webhooks` accepts unauthenticated POST requests from Stripe's servers. The only verification is `stripe.webhooks.constructEvent(body, sig, STRIPE_WEBHOOK_SECRET)`. If the signature check fails, the route returns `400`. No user session is present; all DB operations use the admin client.

### Public portal routes (`/api/portal/*`)

The portal routes handle requests from homeowners, contractors, and property managers via the Nexus property portal. They authenticate via Supabase cookie session (same as dashboard routes). The `shared.ts` normalization helpers do not use the admin client; portal routes use `createClient()`.

## RLS bypass inventory

Every place `SUPABASE_SERVICE_ROLE_KEY` (via `getAdminClient()`) operates on the database:

| Route | Tables written | Reason for bypass |
|---|---|---|
| `POST /api/stripe/webhooks` | `billing_subscriptions`, `profiles`, `payments`, `invoices`, `jobs`, `job_status_history` | Stripe webhooks arrive with no user session; RLS would block all writes |
| `POST /api/match` | None (DB writes use anon client) | Admin client used only for `auth.admin.getUserById()` to look up emails for notification; assignment writes use anon client with `.is('assigned_contractor_id', null)` concurrency guard |
| `GET/POST /api/invoices` | None (DB writes use anon client) | Admin client used only for `auth.admin.getUserById()` to look up client email for invoice notification |

**`auth.admin.getUserById()`** calls in `/api/match` and `/api/invoices` do not write to any table; they retrieve email addresses from Supabase Auth (which is not accessible via RLS-enforced queries). These calls are always in the fire-and-forget email path.

Every RLS policy that uses the `is_admin()` helper (`profiles.role = 'admin'`) is enforced on the **anon** client, not via the admin client.

## Roles and permissions

### Identity

Role is stored in two places that must agree:
- `auth.users.user_metadata.role` вҖ” set at sign-up; used by the middleware redirect logic.
- `profiles.role` вҖ” set during profile auto-creation trigger (`handle_new_user`); used by RLS policies via `is_admin()` and direct column checks.

### Route-level enforcement

Beyond RLS, individual API routes check the caller's role or ownership before performing actions (e.g., `/api/invoices` verifies `contractor_id = user.id` before allowing invoice creation). This is a defence-in-depth layer; RLS is the authoritative enforcement point.

## Contractor matching engine

### Primary pipeline (`service_requests`)

`POST /api/match` with `{ request_id }`:

1. Load the service request; reject if already assigned.
2. Call `match_contractors_by_distance(p_request_id, p_max_results=10)` Postgres RPC (added in migration 017).
3. If the RPC returns at least one row, use those results.
4. **Fallback** (RPC unavailable or returns empty): query `contractor_profiles` directly and score in TypeScript using `scoreContractorMatch()`.
5. Take the top-scored candidate.
6. `UPDATE service_requests SET assigned_contractor_id = $winner, status = 'assigned' WHERE id = $id AND assigned_contractor_id IS NULL` вҖ” the `IS NULL` guard prevents a double-assignment race condition.
7. Insert a `lead_credits` row (best-effort; not fatal if the table is missing).
8. Send match notification emails (fire-and-forget).

**The two paths score identically.** `scoreContractorMatch()` in `lib/business-logic.ts` is the reference implementation; migration 017's RPC implements the same formula in SQL. The RPC is preferred because it runs closer to the data. If the RPC returns a different winner than the TS scorer would for the same input, the RPC wins вҖ” it is canonical when available.

**Scoring weights:**

| Signal | Points |
|---|---|
| Primary trade match | +40 |
| Within service radius (linear decay) | 0вҖ“25 |
| Rating вүҘ 4.5 / 4.0 / 3.5 | +15 / +10 / +5 |
| Verified + compliance docs current | +10 |
| Below max active projects | +10 |

Contractors with expired license or insurance docs are excluded before scoring.

### Legacy pipeline (`jobs`)

`POST /api/match` with `{ job_id }`:

Uses the same `scoreContractorMatch()` scorer but does not call the RPC; it queries `contractor_profiles` directly. For the legacy pipeline, `rating` is not included in the score (always 0) because the job table does not denormalize average rating.

**Implication:** The same contractor may receive a different score for a `job` vs a `service_request` with equivalent parameters. The legacy pipeline is for `jobs` table records only; all new intake creates `service_requests`.

## Fee and invoice lifecycle

### Fee calculation

Fees are calculated at **invoice creation** time by `calculateInvoiceTotals()` in `lib/business-logic.ts`:

```
subtotal   = sum of invoice line item amounts
feeRate    = FEE_RATES[urgency]  вҶ’  routine=0.25, urgent=0.30, emergency=0.35
nexus_fee  = round(subtotal Г— feeRate, 2)
total      = subtotal + nexus_fee
```

The urgency used is the `jobs.urgency` field at invoice creation. If urgency is edited on the request after an invoice is generated, the invoice retains the original fee rate вҖ” there is no recalculation trigger.

The `NEXUS_FEE_RATE_*` env vars in `env.example` are documented as optional overrides but are not read by `lib/business-logic.ts` in the current code. `FEE_RATES` is a hardcoded constant. If you need runtime-configurable rates, add env-var reading to that module.

### Payment flow (invoice path)

```
Contractor creates invoice (draft)
  вҶ’ Contractor sends invoice (status: sent)
  вҶ’ Homeowner pays via Stripe Checkout (/api/stripe/checkout)
      вҶ’ Stripe fires checkout.session.completed
          вҶ’ /api/stripe/webhooks:
              invoices.status = 'paid'
              jobs.status = 'completed'
              job_status_history insert
              receipt emails sent (fire-and-forget)
```

### Payment flow (subscription path)

```
User initiates checkout (/api/stripe/checkout)
  вҶ’ Stripe Checkout session created with mode=subscription
  вҶ’ User completes checkout
  вҶ’ Stripe fires checkout.session.completed
      вҶ’ /api/stripe/webhooks:
          billing_subscriptions upsert (onConflict: stripe_subscription_id)
          profiles.subscription_tier = planId
          profiles.subscription_status = sub.status
```

Subscription renewals arrive as `invoice.paid` events. The webhook handler does not modify subscription records for renewals вҖ” only `checkout.session.completed` bootstraps a subscription record and `customer.subscription.updated/deleted` handles lifecycle changes.

## Stripe webhook idempotency

Stripe may deliver the same event more than once. Current behavior for each event type:

| Event | Idempotent? | Risk |
|---|---|---|
| `checkout.session.completed` (invoice) | Yes вҖ” setting `invoices.status = 'paid'` and `jobs.status = 'completed'` twice is a no-op | None |
| `checkout.session.completed` (subscription) | Yes вҖ” `billing_subscriptions` uses `upsert` with `onConflict: stripe_subscription_id` | None |
| `invoice.paid` | **Partially** вҖ” DB updates are idempotent; receipt emails are not | Duplicate receipt emails possible on retry |
| `customer.subscription.updated/deleted` | Yes вҖ” updates are set-based, not incremental | None |
| `invoice.payment_failed` | Yes вҖ” idempotent status update | None |
| `charge.refunded` | Yes вҖ” idempotent status update | None |
| `account.updated` | Yes вҖ” idempotent status update | None |

**Known gap:** The `stripe_events` table exists in the schema but the webhook handler does not write to it. Adding an event-id deduplication check (`INSERT INTO stripe_events ... ON CONFLICT DO NOTHING`) before processing would make `invoice.paid` fully idempotent and prevent duplicate receipt emails.

## Email delivery

All transactional emails are sent via **Resend** (`lib/email.ts`). Email sends are fire-and-forget in every call site вҖ” wrapped in an IIFE async block, errors are logged but never propagated.

**This is an explicit tradeoff:** UX smoothness (the API response is not delayed waiting for email delivery) at the cost of delivery guarantees (a recipient may never receive an email and will not be notified of the failure). Affected emails:

| Email | Trigger | Consequence of failure |
|---|---|---|
| Job matched вҖ” contractor notification | `POST /api/match` (jobs pipeline) | Contractor misses the assignment; only learns via dashboard |
| Contractor assigned вҖ” client notification | `POST /api/match` (jobs pipeline) | Client unaware a contractor was assigned |
| Invoice paid вҖ” contractor receipt | `invoice.paid` webhook | Contractor has no payment confirmation email |
| Invoice paid вҖ” client receipt | `invoice.paid` webhook | Client has no payment confirmation email |

If delivery guarantees matter for any of these, options are: (a) make the send synchronous and return an error if it fails, or (b) persist a "pending email" record and process it from a retry queue.

## Neon / Postgres status

`lib/env.ts` exports `getDatabaseUrl()` and `getDatabaseUrlUnpooled()`. These functions are not called by any API route or library in the current codebase. The VercelвҖ“Neon integration sets `DATABASE_URL` and `DATABASE_URL_UNPOOLED` automatically, but nothing reads them.

**Decision required:** Remove `DATABASE_URL*` / `POSTGRES_*` / `NEON_DATABASE_URL` from `env.example` and `lib/env.ts` once confirmed unused, or document the specific future use case that justifies keeping them. Leaving them as-is invites new code to start using Neon alongside Supabase, creating a split-database situation with no clear ownership.

## Migration strategy

Schema is applied via `scripts/migrate.mjs`, which POSTs SQL to the Supabase Management API:

```
SUPABASE_PROJECT_REF=... SUPABASE_ACCESS_TOKEN=... node scripts/migrate.mjs
```

The canonical schema source is `scripts/setup.sql`, which is idempotent (`CREATE TABLE IF NOT EXISTS`, `drop policy if exists`, `create or replace function`). Running `setup.sql` twice is safe.

The numbered migration files (`001_*.sql` through `018_*.sql`) are incremental scripts for applying specific changes without re-running the full setup. To apply one:

```
SUPABASE_MIGRATION_FILE=018_search_and_profiles.sql node scripts/migrate.mjs
```

**Why not Supabase CLI?** The Management API approach works without installing the Supabase CLI toolchain in CI and without requiring a local Docker instance. The tradeoff is that migration history is not tracked in Supabase's `supabase_migrations` table.

**Collision risk:** If two PRs each add a file named `019_*.sql`, the first to merge wins and the second conflicts on filename. The mitigation is to coordinate migration numbering via PR description or to use `setup.sql` as the single source of truth and not rely on numbered files for production migrations.

## Failure modes

| Dependency | Failure mode | Current behavior |
|---|---|---|
| Supabase unavailable | All DB reads/writes fail | Unhandled вҖ” routes return 500; middleware session refresh skipped (graceful fallback to `NextResponse.next()`) |
| Supabase anon key missing | `getSupabaseServerConfig()` throws | App fails to start if env is missing at build time |
| Stripe API unavailable | Checkout, subscription, Connect onboarding fail | Handled per-route with try/catch; users see error state |
| Stripe webhook delivery delayed | Events arrive late | DB writes are idempotent; delayed delivery is safe for most events (see idempotency table) |
| OpenAI rate-limited / unavailable | Categorization, chat, insights fail | `/api/automation/categorize-request` returns 500 with `requestId`; user can proceed without AI categorization |
| Resend unavailable / rate-limited | Email delivery fails silently | Fire-and-forget; logged to console only; users are not notified |
| Vercel Blob unavailable | Document uploads fail | Route returns error; upload UI should surface the failure |
| `match_contractors_by_distance` RPC missing | `/api/match` falls back to TS scorer | Graceful; behavior is equivalent to primary path |
| `lead_credits` table missing | Lead credit insert skipped | Best-effort; not fatal |
